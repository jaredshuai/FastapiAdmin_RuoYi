# 当前模块地图

状态：基于提交 `1966c53` 的代码索引与目录核对

## 后端

| 区域 | 当前职责 | 目标约束 |
| --- | --- | --- |
| `backend/app/core/` | 配置、数据库、鉴权依赖、通用 CRUD、缓存、调度、异常、插件发现 | 拆分平台内核与可选公共能力；业务模块不能反向侵入内核 |
| `backend/app/common/` | 枚举、常量和通用声明，`constant.py` 已承载大量跨域常量 | 只保留稳定、无业务归属的声明；业务枚举回归所属模块 |
| `backend/app/config/` | 环境配置、路径和设置加载 | 作为装配与基础设施配置，不导入业务模块 |
| `backend/app/utils/` | 导入、文件、请求、序列化等工具 | 按调用关系归入内核、基础设施或具体模块，禁止继续无边界增长 |
| `backend/app/api/v1/module_system/` | 用户、角色、部门、岗位、字典、参数等系统管理 | 作为身份、组织和系统配置模块，不放具体业务规则 |
| `backend/app/api/v1/module_platform/` | 租户、套餐、菜单、插件、邮件、订单、发票等平台能力 | 收敛为跨业务平台服务；明确哪些能力只允许平台管理员操作 |
| `backend/app/api/v1/module_monitor/` | 在线状态、服务器、缓存等运维观察能力 | 与业务请求链路解耦，禁止把监控作为业务状态源 |
| `backend/app/api/v1/module_common/` | 健康检查、公共接口等 | 保持小而稳定，公开接口必须显式标注安全边界 |
| `backend/app/plugin/module_*/` | 动态发现的扩展控制器 | 作为候选业务模块来源；生产目标改为发布时显式装配，不依赖运行时热重载 |
| `backend/tests/` | 后端测试 | 从“路由存在”升级为业务结果、权限拒绝和租户隔离测试 |

当前典型业务实现由 `controller.py`、`service.py`、`crud.py`、`model.py`、`schema.py` 组成。迁移期允许保留这些文件名，但当前 `service` 可能同时承担用例编排、数据库访问和框架集成，不能把命名直接当作目标边界。Phase 2 先由 ticket 验证无框架用例层、Required Port 和持久化 Adapter；无论是否推广，其他模块都只能通过小型稳定接口对外提供能力，禁止跨模块直接操作 CRUD 或数据表。

当前已确认的平台内核反向依赖包括：`core/dependencies.py` 导入用户/角色 ORM 模型和套餐 `PackageService`。当前装配硬编码包括：`init_app.py` 逐个导入租户、字典、参数、AI WebSocket、全局事件和调度器，`scripts/initialize.py` 逐个导入所有 ORM 模型。这些是 Phase 2 必须消除的存量耦合，不是未来才需要防止的问题。

重点文件归属审查：

| 文件 | 当前规模与职责 | 目标处理 |
| --- | --- | --- |
| `core/ap_scheduler.py` | 约 1547 行，包含调度基础设施和大量行为 | 拆出任务调度 Adapter；模块只声明任务入口，不直接依赖其实现 |
| `core/base_crud.py` | 约 622 行，混合通用 CRUD、租户过滤和数据权限 | 保留稳定持久化接口，将租户/数据策略通过内核接口注入并补负向测试 |
| `common/constant.py` | 约 780 行，聚合跨领域枚举和常量 | 按所有权迁回模块，只保留真正跨业务稳定常量 |

## 当前到目标的映射

| 目标层次 | 当前候选位置 | Phase 2 动作 |
| --- | --- | --- |
| 应用装配层 | `backend/app/init_app.py` 及 Router 注册逻辑 | 收敛为唯一组合根，显式选择并校验模块 |
| 平台内核 | `backend/app/core/` 的契约、事务语义、异常和安全/租户策略接口 | 只保留跨业务稳定机制，反转对业务模块和 ORM 模型的依赖 |
| 必选基础设施 | `core/`、`config/`、`utils/` 中的 Web、数据库、日志和认证实现 | 作为实现内核接口的 Adapter，由装配层选择，不携带业务规则 |
| 可选公共能力 | `core/`、`module_platform/` 中的文件、消息、邮件、任务和缓存实现 | 通过删除测试判定，按真实复用需求拆成可选择的深模块 |
| 平台与业务模块 | `module_system/`、`module_platform/` 和 `plugin/module_*/` | 按领域与数据所有权重组，提供清单、明确的入站入口和跨模块公开接口声明 |
| 独立扩展 | `module_monitor/` 及独立任务或监控进程 | 只有具有独立运行或部署理由时保留在该层 |

目标 Python 导入方向是应用装配层可以依赖被装配组件，业务模块、可选能力和必选基础设施分别依赖平台内核；调用方模块只有边界 Adapter 可以依赖提供方公开接口或声明的公共能力，领域/用例层只依赖自有 Port 与稳定内核 Port。业务模块之间的直接依赖必须形成有向无环图。

## 前端

| 区域 | 当前职责 | 目标约束 |
| --- | --- | --- |
| `frontend/web/src/router/` | 静态壳层、动态菜单转换、路由校验与注册 | 菜单可见性只负责体验，不能替代后端权限 |
| `frontend/web/src/views/` | 系统、平台、任务等页面 | 按业务领域归档；页面不是后端模块边界 |
| `frontend/web/src/api/` | 后端接口封装和类型 | 接口按领域聚合，避免页面内散落请求逻辑 |
| `frontend/web/src/hooks/` | 表格、图表等复用逻辑 | 拆分高复杂度实现，保持页面调用接口简洁 |
| `frontend/web/src/stores/` | 登录态、用户信息和全局状态 | 只保留真正跨页面状态，业务状态归业务模块 |

## 建议的业务模块接口

每个正式业务模块至少应拥有：

- 唯一模块编码和依赖声明。
- 后端 Router、服务接口、数据模型、Alembic 迁移和权限清单。
- 前端页面、API 封装、类型、菜单/路由元数据。
- 操作日志与审计要求。
- 权限、租户、状态流转和主要失败路径测试。
- 安装、升级、停用或删除时的行为说明。
- 不变量、错误模式、配置要求、事务语义和必要的性能约束。

具体目标见[目标架构](../plans/platform-foundation/topics/target-architecture.md)。
