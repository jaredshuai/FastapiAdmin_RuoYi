# 存储与状态边界

状态：初始盘点，待 Phase 0 运行验证

## 关系数据库

- SQLAlchemy 负责异步数据访问，仓库包含 Alembic 配置但 `backend/alembic/versions/` 只有 `.gitkeep`，当前没有有效迁移。
- 依赖中同时包含 MySQL、PostgreSQL 和 SQLite 驱动；当前 Docker 示例默认使用 MySQL 8。
- 请求级会话管理事务，通用 CRUD 集中执行租户和数据权限过滤。
- 启动初始化通过 `metadata.create_all` 建表并导入 JSON 种子，当前不能证明历史版本升级或回滚。
- `tenant_id == 1` 与 `__platform_data_shared__` 共同表示平台共享数据：其他租户可读、只有平台管理员可写；`PluginService.install` 还会让该租户绕过套餐插件范围和付费校验。该组平台特权尚未形成稳定决策。

目标要求：

- 正式环境只声明并验证公司选定的主数据库组合。
- 所有表结构变化必须通过可审查迁移落地，不依赖生产启动时隐式改表。
- 每个业务模块明确数据所有权，跨模块读取通过服务接口或明确的只读查询完成。

## Redis

当前用于缓存、登录会话相关状态、限流、租户/字典/参数缓存、调度锁等。

鉴权路径还存在租户套餐菜单的 60 秒进程内缓存；套餐、菜单或权限变化后没有显式的多实例失效协议。

目标要求：

- 每类 Key 有稳定命名、租户边界、过期策略和清理责任。
- Redis 丢失时区分可重建缓存与不可丢业务状态。
- 禁止把需要强一致和可审计的核心业务状态只保存在 Redis。

## 进程内状态

当前存在权限套餐短期缓存、动态 Router 缓存、应用引用和调度器运行状态。

目标要求：

- 多进程和多实例部署前，逐项证明进程内状态是否允许不一致。
- 热重载、缓存失效和配置变化要有显式传播方式。
- 业务模块不能依赖某个进程“恰好执行过初始化”。

## 文件与配置

- 静态文件、上传文件和环境配置目前分布在后端及 Docker 路径中。
- 生产镜像和配置加载顺序存在已知问题，详见[风险热点](risk-hotspots.md)。
