# 风险热点

状态：基于提交 `1966c53` 的初始评估；按优先级处理

## P0：扩展业务前必须解决

### 权限通配符绕过

`backend/app/core/dependencies.py::AuthPermission.__call__` 把接口声明中的 `*` 或 `*:*:*` 直接解释为放行。发票模块已有多个接口使用 `AuthPermission(["*:*:*"])`，导致普通登录用户可能绕过预期操作权限。

停止条件：删除含混语义，超级管理员只由可信身份字段判定，并建立完整负向权限测试。

### 生产配置和默认安全值

- `main.py run --env=prod` 设置环境的时机晚于全局 `settings` 导入。
- 默认环境、调试、文档、CORS 和密钥配置不适合作为生产回退值。
- Docker 示例没有完整注入所有生产安全配置。

停止条件：生产环境缺少机密或关键配置时直接启动失败，且配置来源在导入应用前确定。

## P1：底座冻结前解决

### 容器产物不自包含

当前后端 Dockerfile 没有把完整应用代码复制进镜像，Compose 依赖 `../backend:/home` 绑定挂载。删除生产挂载后镜像无法独立运行。

### 测试可能产生假绿

- 后端大量路由测试只验证不是 404，测试辅助函数会吞掉部分异常。
- 缺少普通用户、无权限、停用角色、套餐收缩和跨租户负向测试。
- 前端测试主要是枚举导入 smoke，未覆盖动态路由和关键页面交互。

### CI 位置与行为

- 既有前端工作流位于子项目路径，根仓库可能不会执行。
- 前端 lint 脚本默认修改文件，不适合作为只读 CI 门槛。
- `.auto-import.json`、`auto-imports.d.ts` 和 `components.d.ts` 继续作为忽略的生成物；当前工作树已增加 `gen:auto-imports` 并在隔离干净副本中验证 `lint:check`、`build` 可自动生成。剩余风险是该修正尚未提交且根目录 CI 尚未固化执行顺序。
- 仓库锁定 `pnpm@9.15.3`，而开发机可能安装更高版本的全局启动器；CI 必须断言实际版本，不能因为高版本迁移告警就提前搬走 pnpm 9 仍需读取的 `pnpm.overrides`。

### 管理员能力保护不一致

邮件配置服务声称只允许超级管理员，但创建、更新、删除与页面、详情、测试的保护方式不一致。

### 当前代码违反目标模块依赖方向

- `core/dependencies.py` 直接导入用户/角色 ORM 模型和套餐 `PackageService`。
- `init_app.py` 和 `scripts/initialize.py` 逐个点名模块、服务、WebSocket 和模型。

停止条件：以架构检查记录全部存量违规；Phase 2 通过内核拥有的窄接口和唯一组合根消除它们，G5 转为严格失败。

### 零迁移基线

`backend/alembic/versions/` 只有 `.gitkeep`，启动依赖 `metadata.create_all`。当前无法验证从上一版本升级、迁移顺序和回滚，不满足 G3。

### 插件市场与发布时装配语义冲突

系统已有租户插件市场、安装/卸载/启停接口和租户插件数据，但当前动态路由发现不读取 `TenantPluginModel.enabled`：这些接口只改变租户记录，实际不控制代码或 Router 装卸。产品问题已收窄为“保留能力目录/租户授权入口，还是下线”；禁止在模块改造中静默删除接口或数据，也不能继续让界面暗示它能热装卸代码。

## P2：随模块化演进处理

- `frontend/web/src/hooks/core/useTable.ts` 和 `useChart.ts` 复杂度过高。
- `backend/app/core/base_crud.py` 的条件构造和权限策略承担过多职责。
- 当前动态插件热重载与发布时显式装配的目标并存；Phase 2 需要把热重载限定为开发或兼容能力，并确保生产路径不依赖旧路由移除和多进程运行时同步。
- `reload_dynamic_router` 分别构建挂载 Router 和返回 Router，二者不是同一实例；在 Phase 2 移除生产依赖前不得把其返回值作为实际装配结果。
- README 描述的移动端和文档子项目与实际文件状态存在漂移。

## 风险处理原则

- 安全问题先写失败测试，再修实现。
- 不因为“RuoYi 也有类似功能”就假设当前语义正确。
- 每个 P0/P1 修复都要留下命令、测试或容器 smoke 证据，并回写路线图。
