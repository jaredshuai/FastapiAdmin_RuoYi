# 发布时装配模块，暂不建设运行时热插拔

日期：2026-07-17

状态：Accepted

## 背景

当前代码既有静态 Router 注册，也有基于目录扫描的动态插件发现和热重载。运行时卸载 Router、迁移、事件处理器和任务，在多进程或多实例环境下需要处理状态传播、失败隔离和回滚，复杂度明显高于当前业务需求。

同时，系统已经存在租户可见的插件市场、`install/uninstall/toggle/my_plugins` 接口，以及 `platform_plugin`、`platform_tenant_plugin` 数据。这里的“安装”目前只修改租户授权和启用记录；动态目录扫描不读取 `TenantPluginModel.enabled`，所以它并不控制 Router 或 Python 代码装卸。命名上容易与运行时代码安装混淆，不能在模块化改造中静默删除或改变语义。

RuoYi-Vue-Plus `v5.6.2` 的主要模块化方式是构建期聚合：`ruoyi-admin` 作为装配入口，`ruoyi-common` 提供细分公共能力，`ruoyi-modules` 保存业务模块，`ruoyi-extend` 保存独立扩展。该结构证明清晰的模块边界不依赖运行时热插拔。

## 决策

- 平台采用发布时模块化单体，模块集合在构建和部署时确定。
- 建立唯一应用装配层，依据显式模块清单校验依赖并注册 Router、迁移、权限、菜单、事件、任务和前端入口。
- 模块变更通过新制品和应用重启生效，不承诺在运行中的进程内安装、卸载或热替换代码。
- 模块停用表示下一次装配时不包含该模块；租户套餐收缩只影响能力授权，不等同于卸载代码。
- 现有动态插件发现可以在过渡期保留为开发工具或兼容入口，但不作为正式模块契约；生产路径必须可关闭它。
- 发布时装配决定“制品是否包含代码”，租户套餐或插件记录决定“租户是否拥有和启用能力”，两者必须使用不同状态和术语。
- 本决策不擅自决定现有插件市场接口和数据表的产品归宿；后续只需在“保留为能力目录/租户授权入口”与“下线”之间形成产品决策。在此之前保留数据和外向语义，不执行破坏性迁移。

## 影响

- 模块依赖、数据库迁移、Router、任务和事件的版本集合可以在发布前完整验证。
- 多进程和多实例不需要同步运行时代码装卸状态，生产故障面更小。
- 在线加入制品中原本不存在的第三方模块需要发布新版本，首期接受这一限制。
- 模块清单仍需声明版本、依赖和生命周期，用于装配校验、升级和停用，而不是实现运行时插件市场。
- `reload_dynamic_router` 不能进入生产装配路径；现有市场 API 是否保留为能力目录/租户授权入口，仍由[未决问题](../open-questions.md)约束。

## 重新评估条件

只有出现无需发布即可安装第三方模块的明确产品需求，并且能够给出签名校验、迁移隔离、多实例一致性、失败回滚和运维责任方案时，才重新评估运行时热插拔。
