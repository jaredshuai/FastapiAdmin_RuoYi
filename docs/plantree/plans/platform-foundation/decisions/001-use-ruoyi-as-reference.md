# 把 RuoYi-Vue-Plus 作为参考项目，而不是迁移源

日期：2026-07-17

状态：Accepted

## 背景

项目目标是建设一个由账户权限控制页面和操作、能够持续承载公司业务需求的平台。RuoYi-Vue-Plus 在企业后台、权限、租户、数据范围、运维和开发工具方面有丰富经验，但其实现深度绑定 Java 和 Spring 生态。项目主要维护技术栈是 Python/FastAPI。

## 决策

- 保留 FastAPI + Vue 技术路线。
- 使用 RuoYi-Vue-Plus 作为能力清单、业务规则、边界案例和验收场景的参考。
- 按业务优先级逐项采用或适配能力，不进行全量 Java 到 Python 翻译。
- 当前系统已有能力先修正确性和生产可靠性，再讨论功能扩张。

## 影响

- 可以保持与维护者技能匹配，并拥有完整实现控制权。
- 每项能力都需要在 FastAPI 语境下重新设计接口和测试，不能因为 RuoYi 已实现就视为本项目已成熟。
- 不保证与 RuoYi 的 API、数据表、插件或发布节奏兼容。
- RuoYi 上游安全修复和行为变化作为风险输入，仍需独立判断是否适用。

## 重新评估条件

如果公司建立稳定 Java 团队，或者出现必须直接使用 RuoYi 生态组件且 Python 重建成本明显更高的核心需求，可以创建新决策重新比较“直接使用 RuoYi”和“继续建设当前平台”。
