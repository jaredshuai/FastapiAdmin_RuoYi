# RuoYi-Vue-Plus 对标方法

角色：参考项目使用规则

状态：已接受战略，能力优先级待业务确认

相关决策：[001-use-ruoyi-as-reference.md](../decisions/001-use-ruoyi-as-reference.md)、[002-release-time-module-assembly.md](../decisions/002-release-time-module-assembly.md)

## RuoYi 作为哪类老师

- 产品老师：企业后台通常需要解决哪些管理和运维问题。
- 业务规则老师：角色、菜单、租户、数据范围、停用、删除和审计有哪些边界情况。
- 验收老师：把成熟项目的行为转化为本项目的正向和负向测试。
- 架构参考：观察模块归属和扩展点，但用 FastAPI、SQLAlchemy、Vue 的惯用方式实现。

RuoYi 不是逐文件翻译源，也不是必须追平的功能清单。

## 固定参考源

参考源码以 Git submodule 保存在仓库根目录的 `references/`，具体获取和更新方法见 [`references/README.md`](../../../../../references/README.md)。

| 参考项 | 固定版本 | 用途 |
| --- | --- | --- |
| `references/ruoyi-vue-plus/` | `v5.6.2` | 后端权限、租户、数据范围、运维和模块边界 |
| `references/plus-ui/` | `v5.6.2-v2.6.2` | 菜单、路由、按钮权限和页面交互 |

两个参考仓库只读并分别建立代码索引，不进入本项目的构建、测试和发布范围。对标结论必须记录所查源码路径和固定提交，不能只引用分支名称。

## 模块化对标结论

采用并适配 RuoYi-Vue-Plus 的结构角色：

- 以应用装配层对应 `ruoyi-admin`，只负责选择和组合模块，不放业务规则。
- 以平台内核对应 `ruoyi-common-core` 的最小稳定契约，不把全部 `ruoyi-common-*` 压成同一层。
- 把 `mybatis/redis/satoken/security/log/web/json` 等视为 RuoYi 的基础设施层证据；在本项目中按“目标制品是否必需、是否实现内核接口”重新判定必选 Adapter，不能机械复制名称。
- 以 `oss/mail/excel/sms/sse/websocket/idempotent/ratelimiter` 等为可选能力参考，只有声明依赖的模块才装配。
- 以平台与业务模块对应 `ruoyi-modules`，每个模块拥有清单、公开接口和数据所有权。
- 以独立扩展对应 `ruoyi-extend`，只有存在独立运行或部署理由时才进入该层。
- 用 Python 依赖锁定、模块清单和架构测试替代 Maven 聚合与 BOM，不复制 Java 目录和构建方式。
- 默认采用发布时装配，不把运行时热插拔当作模块化成立的前提。

## 单项能力对标模板

每次只评估一个能力，并记录：

1. 公司要解决的真实问题和用户角色。
2. RuoYi-Vue-Plus 对该问题的行为、边界和官方版本/提交证据。
3. 当前 FastapiAdmin 已有能力和已知缺陷。
4. 选择：采用、适配、推迟或拒绝。
5. FastAPI 目标接口、数据所有权和失败模式。
6. 权限、租户、审计、迁移、性能和回滚验收。
7. 实现与验证证据。

## 初始优先级

| 优先级 | 能力 | 当前策略 |
| --- | --- | --- |
| P0 | 登录、角色、菜单、API 权限、数据权限、租户边界 | 先修正确性，再扩展功能 |
| P0 | 配置、密钥、日志、测试、迁移、生产容器 | 建立生产底线 |
| P1 | 字典、参数、文件、通知、定时任务、幂等 | 按真实业务场景深化接口 |
| P1 | 代码生成 | 生成项目标准模块，不复制旧模板缺陷 |
| P2 | SSO、第三方登录、SSE/WebSocket | 账户来源和实时需求明确后评估 |
| P2 | 工作流 | 首个需要审批的业务先验证固定状态机是否足够 |
| Deferred | 多数据库并行、微服务、低代码、全量移动端 | 没有明确需求前不投入 |

## 跟踪节奏

- 以项目里程碑和安全修复为触发，不追逐 RuoYi 每个提交。
- 对标时记录官方分支、版本或提交，避免“印象中的 RuoYi”。
- RuoYi 新增能力不会自动进入本项目路线图，必须经过业务价值和维护成本判断。
