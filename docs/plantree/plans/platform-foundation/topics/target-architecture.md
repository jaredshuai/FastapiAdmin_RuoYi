# 目标架构

角色：实现约束

状态：提议，待计划确认后作为 Phase 2 输入

## 总体形态

采用模块化单体：一个后端部署单元、一个 Web 前端、共享数据库基础设施，但代码和数据所有权按业务领域隔离。平台规模增长不自动等于微服务；只有独立团队、独立发布、显著负载差异或故障隔离需求出现时才拆分。

生产环境默认采用发布时模块装配：每个制品包含的模块集合在构建和部署时确定，启动时完成清单校验和注册。首期不支持在运行中的进程内安装、卸载或热替换代码，具体决策见[发布时模块装配](../decisions/002-release-time-module-assembly.md)。

## 目标分层

参考 RuoYi-Vue-Plus 的装配层、内核、基础设施、可选能力、业务模块和独立扩展分工，但使用 Python/FastAPI 的模块和接口实现：

| 层次 | 职责 | 依赖约束 |
| --- | --- | --- |
| 应用装配层 | 创建 FastAPI 应用，选择模块，执行清单校验，依序注册迁移、Router、事件和任务 | 可以依赖所有被装配模块，但不承载业务规则 |
| 平台内核 | 模块契约类型与纯校验函数、安全与租户策略接口、事务语义、异常等跨业务稳定机制 | 不依赖基础设施 Adapter、业务模块或可选能力，也不反向调用装配层 |
| 必选基础设施 | Web、数据库会话、日志、安全认证等当前制品不可缺少的 Adapter | 实现平台内核定义的接口，不依赖业务模块 |
| 可选公共能力 | 文件、消息、邮件、Excel、任务、限流等可移除的深模块 | 依赖平台内核和必要基础设施；不能包含具体业务模型 |
| 平台与业务模块 | 身份、组织、权限、租户配置以及公司业务领域 | 依赖平台内核、声明的可选能力和其他模块公开接口；只允许导入白名单中的框架边界类型 |
| 独立扩展 | 监控、独立任务服务等具有单独运行或部署理由的能力 | 通过公开接口或事件协作，不反向侵入平台内核 |

建议的目标物理结构如下；Phase 2 先迁移一个参考模块验证结构，不进行无验证的全量改名：

```text
backend/app/
  host/                 # 应用装配层和唯一组合根
  core/                 # 平台内核
  infrastructure/       # 必选基础设施 Adapter
  capabilities/<name>/  # 可选公共能力
  modules/<domain>/     # 平台与业务模块

frontend/web/src/
  app/                  # 应用启动、Router 和全局状态装配
  shared/               # 无业务归属的稳定复用能力
  modules/<domain>/     # 页面、API、路由、类型和权限元数据
```

目录名可以在实现前按现有代码成本调整，但分层职责、依赖方向和唯一装配入口不能省略。

## 平台内核、必选基础设施与可选能力

平台内核只承载跨业务稳定机制和由内核拥有的接口：

- 模块契约、事务语义和异常类型。
- 身份会话、权限、租户和数据范围所依赖的稳定策略接口与请求上下文。
- 模块清单契约类型、无副作用的校验函数、迁移描述和生命周期协议；校验执行、实例化和注册编排不属于内核。
- 审计上下文、事件发布和任务注册等基础接口。

必选基础设施是每个目标制品都必须装配、用于实现内核接口的 Adapter，例如 FastAPI Web、SQLAlchemy 数据库会话、日志和选定的认证实现。Redis 是否属于必选基础设施取决于 Phase 0 的正式运行基线；如果允许无 Redis 制品启动，它就是声明依赖的可选 Adapter，不能因参考项目使用 Redis 就直接判为必选。

可选公共能力必须通过“删除测试”：从制品移除后，平台内核和无依赖的业务模块仍能启动，只损失清单中声明的能力。无法满足该条件、被所有制品强制依赖的能力，应归入必选基础设施；包含具体业务状态或数据表的能力，应归入平台或业务模块，而不是放进 `common`。

身份、组织、角色、菜单和租户配置面作为默认平台模块交付；文件、消息、邮件、Excel 和独立任务等能力优先成为可选公共能力。租户归属取决于[未决问题 1](../open-questions.md)：若租户是所有制品必选的业务边界，租户身份、数据过滤和套餐授权所需的稳定策略接口属于内核，租户/套餐 CRUD、配置页面和持久化实现属于默认平台模块并通过 Adapter 接入；若租户可关闭，这些过滤策略应下沉为可装配的 Mixin/Adapter，内核只保留通用请求上下文与策略扩展点。在结论形成前，不把具体租户过滤规则固化进内核。业务状态、业务审批和业务报表不进入内核，`common` 或 `core` 也不能成为无明确归属的代码收容区。

## 业务模块

一个业务需求可以产生多个页面；模块边界按业务概念和数据所有权划分，不按页面文件划分。

迁移期允许保留当前 `controller/service/crud/model/schema` 文件名，但不能把它们直接视为目标边界：`controller` 应收敛为入站 Adapter，具体 CRUD/ORM 属于持久化 Adapter，用例编排不能继续因 `service` 命名而同时承担数据库访问和框架集成。Phase 2 先由 ticket 参考模块验证“领域规则 + 用例 + Required Port + Adapter”边界，细则见[用例与 Port 边界](use-case-and-port-boundaries.md)。不在验证前强制全平台改目录。

每个模块仍需明确入站入口、跨模块公开接口声明和机器可校验的模块清单。没有真实同步调用者时 Provided Interface 可以显式为空；存在公开接口时，不仅要登记可调用入口，还要声明调用方必须知道的不变量、错误模式、配置要求、事务语义和必要的性能约束。

现有 `backend/app/plugin/module_*/plugin.toml` 是模块清单前身。Phase 1 先定义 Pydantic 模型并导出 JSON Schema；Phase 2 再实现旧字段映射和兼容读取。兼容期以新清单为权威；仅当模块尚无新清单时才允许读取 `plugin.toml` 作为降级来源并输出告警。两者冲突时不得静默合并或由读取顺序决定结果。

模块清单至少声明：

- `manifest_schema_version`、`code`、名称、版本、兼容范围和直接依赖。
- 后端 Router 与跨模块 Provided Interface 声明；不存在真实同步调用者时允许显式空列表，不能为满足 Schema 制造公开服务。
- 数据迁移、迁移顺序和数据所有权。
- 菜单、操作/API 权限和默认角色建议。
- 前端组件懒加载入口、API、类型以及引用的菜单/权限编码。
- 发布和订阅的事件、定时任务和外部集成。
- 审计规则、安装/升级/启用/停用行为和验证命令。
- 结构化 `failure_modes`、事务模式和配置要求，以及对应的契约测试入口。

模块内部实现可以复杂，但其他模块和测试看到的接口应当小而稳定。调用方和测试通过同一接口使用模块，不把内部 CRUD、数据模型或私有 Adapter 暴露为跨模块依赖。Schema 可以验证字段、枚举、版本和引用完整性；业务不变量、错误语义和性能承诺必须由公开接口测试证明，不能假称仅靠清单即可自动验证。

## 装配与生命周期

- 应用装配层是唯一组合根，显式选择当前制品启用的模块，不以任意目录扫描结果作为生产契约。
- 平台内核拥有清单 Pydantic 类型、JSON Schema 和无副作用校验函数；应用装配层负责调用校验、选择实现、实例化 Adapter 和执行注册。内核不读取制品目录、不创建应用，也不调用装配层。
- 启动时由装配层校验模块编码唯一、Schema 版本可解析、版本兼容、依赖闭包完整且不存在循环依赖；失败时拒绝启动。
- 按依赖拓扑顺序注册迁移、Router、事件处理器和任务，并保证后端模块、前端入口、菜单和权限清单属于同一个版本集合。
- 前端继续使用后端菜单驱动动态 Router，但构建期必须把 `component_path` 和权限编码与 `import.meta.glob` 可加载组件集合、模块清单声明做交叉校验；版本以同一份模块清单为准，不另外发明前端版本号。
- 模块停用表示下次装配时不注册其入口，不承诺在运行中的多进程内卸载 Python 代码或撤销 Router。
- 现有动态插件发现和热重载能力在 Phase 2 前只视为兼容实现或开发工具，不能定义正式模块接口。

## 依赖规则

必须分别描述 Python 导入方向与运行时注入方向，不能用一条箭头同时表示两种关系。

Python 导入方向由 `import-linter` 或等价工具校验：

```text
host ──> core
     ├─> infrastructure ──> core
     ├─> capabilities ────> core
     └─> modules ─────────> core

modules/<consumer>/application ──> 自有 application/ports + core 稳定 Port
modules/<consumer>/adapters ─────> capabilities 的公开接口
                              └──> modules/<provider>/interfaces

capabilities ──> 经声明的基础设施边界（确有需要时）
modules/<domain>/adapters ──> 经白名单批准的框架边界类型（仅集成例外）
```

- 平台内核禁止导入基础设施 Adapter、可选能力或业务模块。
- 必选基础设施只实现内核接口，不导入业务模块。
- 可选公共能力禁止导入业务模块，且不能拥有具体业务数据表。
- `Request`、`Depends` 等框架集成类型只能进入入站 Adapter，允许列表必须集中、可审查；该例外不包括身份、套餐、权益、审计等具体业务服务或 Adapter。`AsyncSession` 不是纯类型边界，而是对 SQLAlchemy 实现和任意表写入能力的受控泄漏：存量模块可为降低迁移成本暂时保留，但“拿到 Session”不代表拥有所有表，必须配套数据所有权契约测试；ticket 参考模块明确把 Session 收进持久化 Adapter，用例只依赖 `TicketRepository`，再按验证证据决定是否推广仓储/Unit of Work 接口。
- 业务模块的领域/用例层不能直接导入其他业务模块；同步调用由调用方拥有 Required Port，调用方边界 Adapter 只能依赖提供方的公开接口。业务协作 Port 不因复用方便进入平台内核。
- 模块之间只能通过提供方公开接口、领域事件或显式只读查询接口协作；直接依赖必须进入清单，双向同步依赖由 DAG 检查拒绝。
- 禁止跨模块导入 `crud`、`model`、内部 Adapter 或直接写入其他模块的数据表。
- 模块依赖必须形成有向无环图；传递依赖不能替代清单中的直接依赖声明。
- 前端业务模块只能依赖 `shared` 和已声明的公开类型或入口，不能直接导入其他模块的页面和内部状态。

运行时注入方向由应用装配层负责：

```text
host 读取并校验清单
  -> 实例化 infrastructure / provider public interfaces
  -> 实例化 consumer outbound Adapters
  -> 绑定 core 稳定 Port 与 consumer-owned Required Port
  -> 构造 module use cases，并注入 Port 句柄
  -> use cases 只调用 Port，不持有具体业务 Adapter
```

因此“基础设施实现内核接口”表示实现关系，不表示内核导入基础设施；“业务模块使用数据库会话”也不授权其直接调用 `PackageService` 等具体业务服务。平台稳定 Port 由内核拥有；具体业务用例需要的 Required Port 由调用方模块拥有，提供方不反向导入该 Port，由边界 Adapter 完成转换。装配顺序固定为：校验清单与依赖图，按稳定拓扑序处理迁移，再注册 Router、事件、任务和生命周期 hook，最后执行启动 smoke。

## 已知当前耦合与反转方向

以下是基于提交 `1966c53` 的已知迁移障碍，Phase 2 不能从“理想目录已经存在”开始：

| 当前事实 | 目标 seam 与处理方向 |
| --- | --- |
| `core/dependencies.py` 直接导入用户、角色 ORM 模型 | 内核拥有用户身份加载接口，身份模块提供 Adapter；内核不认识业务 ORM 类型 |
| `core/dependencies.py` 直接调用 `PackageService.get_tenant_available_menu_ids` | 内核拥有窄的租户权益接口（暂名 `TenantEntitlementPort`），套餐模块提供 Adapter，由装配层注入；接口形态仍需决策 |
| 请求依赖直接使用 `async_db_session` / `AsyncSession` | 作为数据库实现泄漏单独登记，而不是视为纯框架类型；业务规则不得借此直接写入其他模块拥有的表，导入检查之外还要以迁移/显式 SQL 契约测试约束 |
| `TicketService` 同时依赖 `TicketCRUD`、SQLAlchemy `select`、用户 `UserModel`、输出 Schema 和 `auth.db`，状态流转规则与持久化/跨模块查询混在同一类 | ticket 用例提取窄 `TicketRepository` 与最小身份 Port；SQLAlchemy 和用户接口转换进入边界 Adapter，状态流转由无框架用例测试覆盖 |
| `init_app.py` 的 lifespan、Router 和 WebSocket 注册逐个点名模块 | 模块清单声明 init/shutdown hook、Router 和任务，应用装配层统一注册 |
| `scripts/initialize.py` 导入所有 ORM 模型，启动时通过 `metadata.create_all` 建表 | Phase 1 先建立 Alembic 基线；Phase 2 再由清单声明模块拥有的表和迁移入口 |
| `settings.EVENT_LIST` 和 `SchedulerUtil` 由全局启动逻辑加载 | 事件订阅和任务入口迁入模块清单，装配层统一管理顺序与停用语义 |
| 前端 `ComponentLoader` 对全部 `views/**/*.vue` 使用 `import.meta.glob` | 构建期生成组件路径清单，并与后端菜单、权限编码和模块清单校验 |
| `plugin.toml` 与租户插件市场已存在；当前目录扫描不读取 `TenantPluginModel.enabled`，`install/uninstall/toggle` 只改变租户记录 | 清单演进复用 `plugin.toml`；市场 API 只需在“保留为能力目录/租户授权入口”与“下线”之间决策，不能把当前 `enabled` 误当成运行时路由开关 |

## 权限模型

| 层次 | 回答的问题 | 强制位置 |
| --- | --- | --- |
| 菜单/页面 | 用户是否看到入口 | 后端菜单树 + 前端 Router |
| 操作/API | 用户是否能执行动作 | FastAPI 依赖或服务边界 |
| 数据范围 | 用户能操作哪些记录 | 查询策略和写入约束 |
| 租户/套餐 | 该租户是否拥有此能力和数据 | 权限计算、数据层和模块生命周期 |

权限编码必须来自统一注册表或模块清单。`*:*:*` 不能同时代表“超级管理员身份”和“接口公开给所有登录用户”。

## 模块间协作

优先顺序：

1. 同步服务接口，适合需要立即结果和同一事务的调用。
2. 领域事件，适合通知和可重试的后续处理。
3. 显式只读查询，适合稳定报表场景。

禁止直接实例化其他模块的 CRUD 后修改其表；禁止共享可随意写入的“大公共模型”。

同步协作默认采用“调用方用例 -> 调用方 Required Port -> 边界 Adapter -> 提供方公开接口”。调用方用例层不认识提供方，提供方也不反向依赖调用方。只有身份上下文、租户权益、审计等跨业务稳定策略允许成为内核 Port；业务协作不能为了隐藏循环而上提到 `core`。出现双向同步需要时，必须重划边界、引入独立编排模块或改用事件，不能保留循环导入。详细模板见[用例与 Port 边界](use-case-and-port-boundaries.md)。

跨模块同步调用是否共享请求事务、事件采用事务内还是提交后发布、失败如何回滚，必须在 Phase 2 前形成决策。`tenant_id == 1` 的平台共享数据不变量和套餐权限缓存失效协议同样不能由实现默认值继续隐式决定。

## 架构验收

Phase 2 必须建立自动检查，至少证明：

- 清单缺字段、版本不兼容、依赖缺失或循环依赖时构建或启动失败。
- 平台内核反向依赖业务模块、跨模块导入内部实现时 CI 失败。
- 导入级所有权由架构检查证明；运行时数据所有权另由契约测试证明，至少拒绝模块迁移或裸 SQL 写入其他模块拥有的表。
- 新增、停用或移除参考模块不需要修改平台内核，应用仍能完成构建和启动 smoke。
- 停用模块后，其后端 Router、菜单、权限、前端组件、事件和任务入口不会进入下一次装配结果。
- 数据迁移顺序稳定且可审查，不因文件扫描顺序变化。
- 参考模块的测试只通过公开接口验证主要行为和失败模式。
- 参考模块的领域/用例层不导入 FastAPI、SQLAlchemy、ORM Model 或具体 CRUD；主要状态流转能够使用 Fake Port 在无数据库、Web 和缓存的环境中测试。
- 调用方 Required Port、提供方公开接口和边界 Adapter 的所有权可检查；Fake 与真实 Adapter 的共有行为满足同一契约。
- 全仓跨模块模型导入、裸 SQL 和写表清单已建立，跨模块写入在 G5 前清零；存量 service 只有在仅访问本模块数据时才允许继续作为内部混合职责待迁移项。
- ticket 提交通过实际 Schema 的清单样板，区分 Router、真实 Provided Interface 与内部 use case；内部类重命名不应改变清单。
- 构建期生成模块清单摘要并记录每份清单 hash；启动期重新计算并与不可变制品元数据中的摘要比对，漂移时拒绝启动。

## Phase 2 参考模块

首个参考模块选择 `module_system/ticket`。它规模较小，同时覆盖租户隔离、状态流转、权限编码和对用户模型的跨模块引用，适合优先暴露边界问题。迁移必须验证：

- Router、迁移、权限、菜单、事件、任务和前端入口完全由清单声明，`init_app.py` 与 `scripts/initialize.py` 不新增 `ticket` 特判。
- `assigned_by` 等用户信息通过身份模块公开接口或 `UserIdentityPort` 获取，不跨模块导入用户 ORM relationship。
- 新增、停用或移除 `ticket` 时平台内核不变，停用后的入口不进入装配结果。
- 跨租户读写负向测试、状态转换测试、审计上下文和前后端权限/组件清单校验同时通过。
- ticket 用例通过自有的窄 `TicketRepository` Port 访问数据，SQLAlchemy Adapter 是唯一接收 `AsyncSession` 和访问 ticket ORM 的实现；用例本身不依赖数据库框架。
- 主要用例使用 Fake `TicketRepository` 做无框架测试，Fake 与 SQLAlchemy Adapter 共享可适用的行为契约，数据库约束另由集成测试证明。

首轮范围保持窄：身份读取先实现满足 `assigned_by` 展示所需的最小只读公开接口，不同时设计完整账户体系；前端先验证权限编码，`component_path` 校验在对应构建工具落地后补齐，未补齐前不能宣称参考模块完成 G5。领域事件与定时任务若没有明确业务语义则不加入；事件/任务装配契约改由已有真实调度或事件语义的模块验证，不为了测试清单字段制造伪业务行为。

G5 通过表示全仓跨模块依赖/数据所有权纪律已成立，并由 ticket 证明完整用例隔离模式可行；不表示所有存量 `*Service` 都已迁移为纯用例层。未迁移但仅操作本模块数据的混合 service 必须继续登记为迁移债务；ticket 评审后，新增或发生实质业务修改的模块必须遵循“推广、调整或拒绝”结论提供明确 seam，不能无说明复制旧模式。

## 可拆分信号

只有满足至少一个信号才评估独立服务：

- 模块需要独立扩缩容或资源隔离。
- 模块由独立团队维护并需要独立发布节奏。
- 故障必须隔离，且同步事务已经不是业务要求。
- 数据合规要求独立存储或网络边界。
