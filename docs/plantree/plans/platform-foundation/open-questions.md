# 未决问题

这里只保留尚未决定、会影响方案或阶段验收的问题。

## Phase 0 前建议确认

1. 正式开发和 CI 统一使用 MySQL 还是 PostgreSQL？当前 Docker 示例默认 MySQL。
2. 第一阶段部署目标是单机 Docker Compose、公司内网服务器，还是已有 Kubernetes 环境？

## Phase 2 前必须确认

1. 平台只服务一家公司，还是未来需要多公司/客户租户？这会决定租户模型是核心业务边界还是可关闭能力。
2. 账户来源是本地账户，还是需要企业微信、钉钉、LDAP/AD、OIDC 或其他统一登录？
3. 组织数据权限以部门树为主，是否还存在项目、区域、法人、成本中心等交叉维度？
4. 现有 `platform_plugin`、`platform_tenant_plugin`、插件市场和 `install/uninstall/toggle/my_plugins` 接口，是保留为“制品内能力目录与租户授权入口”，还是下线？当前 `enabled` 不参与 Router 注册，数据可原样保留；需要决定的是外向 API、页面和兼容期，不再把它当成运行时代码装卸方案。
5. 内核读取用户身份、角色菜单和租户套餐权益时，分别使用什么内核拥有的接口及 Adapter？`TenantEntitlementPort` 的输入、返回值、错误模式和缓存责任如何定义？
6. 在已采用“调用方 Required Port + 边界 Adapter + 提供方公开接口”的默认同步模式下，哪些用例必须共享请求级数据库事务？领域事件在事务内还是提交后发布，处理失败是否回滚主业务？
7. `tenant_id == 1` 与 `__platform_data_shared__` 的平台共享数据语义是保留、参数化还是替换？哪些表允许使用该语义？
8. 套餐、菜单、角色或权限变化后，进程内缓存与多实例缓存采用什么失效协议，允许多长时间的不一致？

## Phase 3 前必须确认

1. 第一个真实业务样板是什么？需要能够覆盖至少两种角色和一种数据范围差异。
2. 样板是否需要审批流；如果需要，首期采用固定状态机还是工作流引擎？
3. 样板数据的审计、保留、导出和删除规则是什么？
